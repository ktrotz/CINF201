Best Practices: Building JavaScript/ECMA

  • The very first thing that one should do when building any <script> is to frequently save backup copies of the code. This way, if the code goes sideways and cannot be recovered, only the mots recent few changes need to be rebuilt.
  • As one is building script, build only one or two things at a time. This way, if something goes wrong, it is likely to have happened in creating those few lines.
  • Use comments to temporarily turn off a block of script. Then, reintroduce its code a little at a time. This could mean starting at the top of the a block and working down or it could possibly mean starting on the outside working inward. Always be careful to mind to match the openings to the closings.
    1. A closed comment is made by placing it in-between /* and */.
    2. One can also comment out a single line by placing a // in front of it.
  • When planning out a script, think early about reusability. Should something be a function because it will get a lot of uses? Should something be an array of values that are similar

Exemplar of Testing The Code

Let's say that the goal of the script is to have the user enter five numbers and then to report their arithmetic mean. Plan it out. For example: Will we need to maintain each of the five numbers for later use?

So, let's now make some intentional mistakes. Let's go looking for trouble to learn how to get out of trouble.

Version 0.1: 

<script>

number1 = prompt('Enter your first number:');

</script>

So far, so good, it seems.

Version 0.2: 

<script>

number1 = prompt('Enter your first number:');
number2 = prompt('Enter your second number:');
number3 = prompt('Enter your third number:');
number4 = prompt('Enter your fourth number:');
number5 = prompt('Enter your fifth number:');

alert( 'number1 + number2 + number3 + number4 + number5 );

</script>

And we see nothing working in the script now. But more importantly, there is glaring error that in hiding in plain sight. So, let's comment out some of these new lines. Use /* and */ to isolate blocks of code. If the code runs, then the mistakes are still commented out. If the code doesn't run, the mistake is active script. Divide and conquer.

Version 0.3: 

<script>

number1 = prompt('Enter your first number:');
number2 = prompt('Enter your second number:');
number3 = prompt('Enter your third number:');
/* 
number4 = prompt('Enter your fourth number:');
number5 = prompt('Enter your fifth number:');

alert( 'number1 + number2 + number3 + number4 + number5 ); 
*/

</script>


So far, the code runs. Therefore, the mistake in still in the commented-out part of the code. Let's reintroduce a few more lines of active code.

Version 0.4: 

<script>

number1 = prompt('Enter your first number:');
number2 = prompt('Enter your second number:');
number3 = prompt('Enter your third number:');
number4 = prompt('Enter your fourth number:');
number5 = prompt('Enter your fifth number:');

/*
alert( 'number1 + number2 + number3 + number4 + number5 ); 
*/

</script>

This still works. This guarantees that any mistake must be in the alert( ) line, but it does not guarantee that there is only one mistake. Be careful: magicians exploit the natural human tendency to credit an outcome to a singular cause. Do not make that assumption.

Now that the search is narrowed substantially, a careful inspection of the alert( ) line reveals an open apostrophe without a closed apostrophe. Let's fix that.

Version 0.5: 

<script>

number1 = prompt('Enter your first number:');
number2 = prompt('Enter your second number:');
number3 = prompt('Enter your third number:');
number4 = prompt('Enter your fourth number:');
number5 = prompt('Enter your fifth number:');

// apostrophes paired
alert( 'number1 + number2 + number3 + number4 + number5' ); 

</script>

Oh, right. In our exuberance in locating how the mistake was, we forgot that single-quotes (or apostrophes) generally delimit a literal string. But this wasn't the goal. The goal was to calculate something. So, let's remove the apostrophes.

Version 0.6: 

<script>

number1 = prompt('Enter your first number:');
number2 = prompt('Enter your second number:');
number3 = prompt('Enter your third number:');
number4 = prompt('Enter your fourth number:');
number5 = prompt('Enter your fifth number:');

// apostrophes REMOVED
alert( number1 + number2 + number3 + number4 + number5 ); 

</script>

There is a problem. Let's say that we enter, in order, 10, 20, –5, 3.3333, 6.6667. The total should be 35. And yet, the alert reports "1020–53.33336.6667" We forgot to manage the fact that prompt( ) insists that user responses are text, which then forces the + operator to promote from arithmetic addition to string concatenation. We will need to fix this by putting all 5 prompt( ) functions inside of either a parseFloat( ) (or a parseInt( ) function).

Note: because two of the test values were decimal, parseFloat( ) was selected.

Protip: use the Find & Replace option in your code editor to apply all 5 changes quickly. (Pay attention to whether these are the only prompt( ) functions in your code, in which case just use the Replace All, or else just carefully align the Finder to the beginning of this block and Replace Next for each of the 5 instances. Either way, do not forget to close the parentheses that will be opened this way.

Version 0.7: 

<script>

// prompts now to be parsed as floating-point decimals, 
// parentheses matching checked
number1 = parseFloat(prompt('Enter your first number:'));
number2 = parseFloat(prompt('Enter your second number:'));
number3 = parseFloat(prompt('Enter your third number:'));
number4 = parseFloat(prompt('Enter your fourth number:'));
number5 = parseFloat(prompt('Enter your fifth number:'));

// apostrophes REMOVED
alert( number1 + number2 + number3 + number4 + number5 ); 

</script>

Finally, we have achieved 35, the expected summation for 10, 20, –5, 3.3333, and 6.6667. But does our script work? That depends. Technically, it does what we told it to do. However, way back at the beginning, the goal was to find the arithmetic mean of 5 user-provided numbers. We haven't told the script to do that part yet. So...

Version 0.8: 

<script>

// prompts now to be parsed as floating-point decimals, 
// parentheses matching checked
number1 = parseFloat(prompt('Enter your first number:'));
number2 = parseFloat(prompt('Enter your second number:'));
number3 = parseFloat(prompt('Enter your third number:'));
number4 = parseFloat(prompt('Enter your fourth number:'));
number5 = parseFloat(prompt('Enter your fifth number:'));

// apostrophes REMOVED, tested, it works, this line is no longer needed as is.
// alert( number1 + number2 + number3 + number4 + number5 ); 
sum = ( number1 + number2 + number3 + number4 + number5 ); 
mean = sum/5;

output = 'The artihmetic sum of ';
output += number1 + ', ';
output += number2 + ', ';
output += number3 + ', ';
output += number4 + ', and ';
output += number5 + ' is ' + mean + '.';
alert(output);

</script>

Finally, the script works and meets its goal. Let's review the kind of errors that we have encountered. Keep in mind that there are Three Types of Programming Errors.

  • syntax errors: basically a typo, although usually it's some functional punctuation missing, included, unpaired, or unescaped.
  • runtime errors: mistakes that generally happen during the script execution; often, this is when a user event is out of bounds for the script that is written.
  • logic errors: when a script technically works flawlessly but doesn't address the scripting goals.

Notice that we had a syntax error and mutliple logic errors. And we still have the opportunity for runtime errors because nothing prevents a user from entering a string literal that cannot be parsed as a floating-point decimal.
